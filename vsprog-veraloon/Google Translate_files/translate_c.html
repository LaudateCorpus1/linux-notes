This post was last edited by Simon21ic at 2013-3-5 09:40

1. Data update (including program update, chip configuration data update, programming script update, current chip data update)
press and hold the button, connect to the PC via USB, and a U disk MSCBoot, this is a bootloader

LOST.DIR that was open sourced last time, just to prevent the android system from automatically creating this directory, currently MSCBoot does not support adding new files.
config.bin is the configuration data of each chip. It currently includes nearly 20 types of chips, 165K of data, and a maximum size of 240K. It should be possible to add a lot of chips.
program.bin is the program of the programmer, using 175K, and the maximum is 224K , It can also increase the support of many chips. Readme.txt
can 54
target.bin is the target data.
scripts.txt is the programming script. The

update method is very simple, just copy the file. Note that, except for scripts.txt, the size of the copied file cannot exceed the original size.

2. Configure the programming script
Open scripts.txt and enter the following (automatically generated by the host computer will be provided in the future)

  2. init
  3. beeper.init
  4. beeper.on
  5. delay.delayms 500
  6. beeper.off

  8. slot 0
  9. c stm32f1_md
  10. ms
  11. tvcc.set 3300
  12. prepare
  13. o ef
  14. o wf
  15. o vf

  17. auto_program
Copy code

After the first paragraph just let on to initialize the buzzer 500ms
after selecting the slot 0, and configure the chip type operations to be performed, etc.
Finally, call auto_program script work

since seen the slot 0, it is estimated guess This can support multiple simultaneous programming, but it is only supported by the current architecture design, and the corresponding hardware
has not been made. The current architecture can support multiple simultaneous programming, and even each channel can be set with a different chip to perform differently. The operation script of the computer, burn different programs

The entire software architecture mainly considers easy maintenance and reduces maintenance costs. Performance is not the main consideration, but the measured performance is also pretty good.
When the USB is not turned on, the average download speed of stm32 is about 25KB/s (including all operations).
The above test data uses the chicken blood USB host port of SONY VAIO Z29, the reason for chicken blood: USB2.0 port, but there is no 1ms interval limit, the speed is closer to offline.

The cost of shell mold opening is being evaluated. If it goes well, it is expected to be put on the market before summer vacation. The estimated price is 239 (domestic price).
I tested the STM32F103_MD, and the chips that are expected to support offline programming are as follows:
stm32f1, stm8
will be slowly tested in the future (according to user needs) to support the following:
avr8, spi_dataflash, iic_eeprom, Xintang CortexM0, lpc1000
and some less commonly used or not available The test conditions of the chips need the help of the user to test, the list is as follows:
psoc1, at89s5x, at91sam3, c8051f, stm32f2, stm32f3, stm32f4, stm32l1, hcs08, hcs12, msp430_JTAG, lm3s, kinetis, microwire_eeprom
, and some of them have test conditions. Will do some tests.
If you need to customize, you can develop:
avr32, avrxmega

Of course, untested chips may not be able to support. The support list is subject to the one given at the time of the final release.
